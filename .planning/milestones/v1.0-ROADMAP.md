# Milestone v1.0: MVP

**Status:** SHIPPED 2026-02-21
**Phases:** 1-6
**Total Plans:** 27

## Overview

claude-history delivers universal queryable access to Claude Code's session history through a single Rust binary. Built bottom-up following the Cargo workspace dependency graph: core types and ingestion pipeline, then search and CLI, then HTTP API and daemon, then real-time capabilities, then artifact layer, and finally version monitoring. 16,688 LOC (15,969 Rust + 719 SQL), 148 tests, 39 source files, 6 SQL migrations.

## Milestone Summary

**Key Decisions:**

- Rust over TypeScript for single-binary, zero-runtime-dependency distribution with serde enum system mapping directly to JSONL discriminated records
- 3-crate Cargo workspace (core/store/server) for separation of concerns
- serde(flatten) overflow on every struct for never-discard schema drift detection
- Byte-offset incremental sync exploiting append-only JSONL semantics
- Tool result matching via buffered assistant message (tool_use in assistant, tool_result in next user message)
- File content reconstruction via operation replay (writes + edits in timestamp order)
- version_history table renamed from schema_versions to avoid collision with migration tracker
- progress_events and queue_operations tables dropped in migration 005 (zero semantic value, ~70% of database size)

**Issues Resolved:**

- FTS5 column names must match source table columns for external-content rebuild
- rusqlite 0.37 pinning required by tokio-rusqlite 0.7.0 dependency chain
- axum 0.8 path parameter syntax ({id} not :id)
- CancellationToken lifetime requirements for tokio::spawn in graceful shutdown

**Technical Debt Incurred:**

- DECOMP-03/DECOMP-04 semantics changed: progress_events and queue_operations tables dropped in migration 005 — decomposer now skips INSERT for these record types but preserves session upsert and drift logging
- Phase 1 SC-1 references queue_operations and progress_events tables that no longer exist post-migration 005
- REQUIREMENTS.md traceability table never updated during execution — all 102 requirements show "Pending" despite all phases passing verification
