Implement the JSONL parser per the research "Pattern 6: Byte-Offset Tracking with Partial-Line Safety" and "Pitfall 1: Byte Offset Drift".

**Key types:**

```rust
pub struct ParseResult {
    pub records: Vec<(JSONLRecord, u64)>,  // (record, byte_offset_of_line_start)
    pub warnings: Vec<ParseWarning>,
    pub new_offset: u64,                   // safe offset for next sync
    pub lines_parsed: usize,
    pub lines_skipped: usize,              // empty lines
    pub lines_failed: usize,               // malformed lines
}

pub struct ParseWarning {
    pub line_number: usize,       // 0-based from start of file (or from offset)
    pub byte_offset: u64,         // absolute byte offset of the line
    pub error: String,            // serde error message
    pub raw_line_preview: String, // first 500 chars of the line
}
```

**parse_jsonl function:**

`pub fn parse_jsonl(path: &Path, from_offset: u64) -> Result<ParseResult, ParseError>`

Implementation:
1. Open file, get file length from metadata
2. If from_offset >= file_length, return empty ParseResult (no new data)
3. Seek to from_offset
4. Read lines with BufReader::lines()
5. For each line:
   - Track byte length: `line.as_bytes().len() + 1` (the +1 is for the newline stripped by lines())
   - Skip empty/whitespace-only lines (increment offset, increment lines_skipped)
   - Attempt `serde_json::from_str::<JSONLRecord>(&line)`
   - On success: push to records with the line's start offset
   - On failure: log via `tracing::warn!` with file, line number, offset, error; push to warnings; increment lines_failed
   - Advance current_offset by line byte length
6. After loop: clamp `current_offset` to `min(current_offset, file_length)` to handle missing trailing newline
7. The `new_offset` in ParseResult is `current_offset`

[CORE-06, CORE-07]

IMPORTANT byte-offset safety considerations:
- Only track the offset PAST lines that successfully parsed or are definitively empty. This means if the LAST line fails (could be a partial line from an active write), the offset should still advance past it IF the line had a newline terminator. If it's the last line with no newline (detected by current_offset > file_length), do NOT advance past it — set new_offset to the start of that failed line so it's re-attempted on next sync.
- Actually, the simpler approach per the research: always advance offset past every line (parsed or not), then clamp to file_length. Failed lines at EOF that are partial will be re-attempted because clamping means the offset doesn't go past file end. But mid-file malformed lines (which DO have newlines) are skipped permanently with just a warning — this is correct behavior per CORE-07.

Use `thiserror` for `ParseError`.
  </action>
