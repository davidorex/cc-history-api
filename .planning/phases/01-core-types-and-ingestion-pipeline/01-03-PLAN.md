Implement the decomposition engine that takes parsed records and inserts normalized rows into SQLite tables. All functions take a `&rusqlite::Transaction` reference for atomic batch writes. [DECOMP-06]

The `session_id` parameter is passed from the sync layer (extracted from the JSONL filename) and used for lightweight records (summary, file-history-snapshot) that lack a sessionId field.

**Top-level dispatcher:**

```rust
pub fn decompose_record(
    record: &JSONLRecord,
    session_id_from_file: &str,
    tx: &rusqlite::Transaction,
) -> Result<DecomposeResult>
```

Returns a `DecomposeResult { rows_inserted: usize, overflow_fields: usize }` for stats.

**Per-type decomposition:**

1. `decompose_user(r: &UserRecord, tx)` [DECOMP-01]:
   - Upsert session: `INSERT OR IGNORE INTO sessions (session_id, project_path, first_seen_at, version, slug, git_branch) VALUES (?, ?, ?, ?, ?, ?)` using base.session_id, base.cwd, base.timestamp, base.version, base.slug, base.git_branch
   - Insert message: `INSERT OR IGNORE INTO messages (uuid, session_id, type, timestamp, parent_uuid, is_sidechain, user_type, cwd, git_branch, version, slug, agent_id, is_meta) VALUES (...)` with type='user'
   - Decompose message content: iterate over content blocks (or handle plain string case)
     - For MessageContent::Text(s): insert single message_content row with block_type='text', text_content=s, block_index=0
     - For MessageContent::Blocks(blocks): iterate with enumerate, call decompose_content_block for each
   - Upsert agent if agent_id is present: `INSERT OR IGNORE INTO agents (agent_id, session_id, first_seen_at) VALUES (?)`
   - Log overflow via drift.rs

2. `decompose_assistant(r: &AssistantRecord, tx)` [DECOMP-02]:
   - Upsert session (same as user)
   - Insert message with type='assistant', plus model, stop_reason, request_id from inner message
   - Decompose content blocks (assistant always has Vec<ContentBlock>):
     - For each ContentBlock::Text: insert message_content with block_type='text', text_content
     - For each ContentBlock::Thinking: insert message_content with block_type='thinking', text_content=thinking, thinking_signature=signature
     - For each ContentBlock::ToolUse: insert message_content with block_type='tool_use', tool_use_id=id, tool_name=name, tool_input=serde_json::to_string(&input); ALSO insert into tool_executions table (message_uuid, tool_use_id, tool_name, input_json)
     - For each ContentBlock::ToolResult: insert message_content with block_type='tool_result', tool_use_id, text_content=content.to_string(), is_error
   - Insert token_usage if message.usage is Some:
     - input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens, service_tier
     - cache_creation_json: serialize cache_creation Value to string if present
     - extra_json: serialize the overflow HashMap from UsageStats if non-empty
   - Upsert agent if agent_id present
   - Log overflow from both AssistantRecord.overflow and AssistantMessage.overflow

3. `decompose_progress(r: &ProgressRecord, tx)` [DECOMP-03]:
   - Upsert session
   - Insert into progress_events: uuid=base.uuid, session_id=base.session_id, timestamp=base.timestamp, data_type (extract from r.data["type"] as string, default to "unknown"), data_json=serde_json::to_string(&r.data)
   - Log overflow

4. `decompose_queue_operation(r: &QueueOperationRecord, tx)` [DECOMP-04]:
   - Insert into queue_operations: session_id, operation, timestamp, content
   - Log overflow (no session upsert needed — queue ops have session_id but incomplete session info)

5. `decompose_system(r: &SystemRecord, tx)`:
   - Upsert session
   - Insert into system_events: uuid=base.uuid, session_id=base.session_id, timestamp=base.timestamp, subtype, level, duration_ms, content, extra_json (serialize overflow if non-empty, plus fields like hook_count)
   - Upsert agent if agent_id present
   - Log overflow

6. `decompose_summary(r: &SummaryRecord, session_id_from_file: &str, tx)`:
   - Insert into summaries: session_id=session_id_from_file, summary=r.summary, leaf_uuid=r.leaf_uuid
   - Log overflow

7. `decompose_file_history_snapshot(r: &FileHistorySnapshotRecord, session_id_from_file: &str, tx)`:
   - For Phase 1, just log that we encountered it. The spec tables don't include a file_history_snapshots table, but we should not silently drop it. Options: (a) store in a generic catch-all, (b) log as info. Choose (a): create a minimal row somewhere or just skip with tracing::debug. The spec success criteria don't mention this table, so skipping with a debug log is acceptable. Log any overflow.

Implement the overflow-to-drift-log function per research "Pattern: Schema Drift Logging" and requirement DECOMP-05.

**Function:**

```rust
pub fn log_overflow(
    version: &str,
    record_type: &str,
    overflow: &HashMap<String, serde_json::Value>,
    tx: &rusqlite::Transaction,
) -> Result<usize>
```

Returns the number of new drift entries logged.

For each (field_name, value) in the overflow HashMap:
- Truncate sample_value to 500 chars: `let sample = value.to_string(); let truncated = if sample.len() > 500 { &sample[..500] } else { &sample };`
- Insert: `INSERT OR IGNORE INTO schema_drift_log (field_name, record_type, version, sample_value, source_context) VALUES (?1, ?2, ?3, ?4, ?5)` where source_context = `format!("overflow capture from {} record v{}", record_type, version)`
- The UNIQUE(field_name, record_type, version) constraint means each unique combination is logged only once — subsequent occurrences are ignored
- Log at DEBUG level each new field discovered, INFO level for summary count

[DECOMP-05, STORE-04]

Also provide a convenience wrapper that the decomposer calls:

```rust
pub fn log_record_overflow(
    record: &JSONLRecord,
    tx: &rusqlite::Transaction,
) -> Result<usize>
```

This extracts version, record_type, and overflow from the record and calls log_overflow. For records with multiple overflow maps (e.g., AssistantRecord has record-level overflow AND AssistantMessage overflow AND UsageStats overflow), call log_overflow for each, using descriptive record_type values like "assistant", "assistant.message", "assistant.message.usage".
  </action>
  <verify>
Unit tests:
1. Create a HashMap with 2 unknown fields, call log_overflow, query schema_drift_log — 2 rows exist with correct field_name, record_type, version, sample_value

3. Call log_overflow with a value > 500 chars — sample_value is truncated
4. Call log_overflow with empty HashMap — no rows inserted, returns 0
