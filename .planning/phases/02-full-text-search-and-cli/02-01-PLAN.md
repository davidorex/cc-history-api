<objective>
Create the FTS5 full-text search infrastructure and the complete store-layer query API that all CLI subcommands will consume.

Purpose: Phase 2 CLI subcommands need a query layer between the CLI handlers and raw SQL. This plan builds that layer: FTS5 indexing for search, parameterized query builders for filtering, SQL aggregation for stats, and data retrieval for export. It also integrates FTS rebuild into the existing sync pipeline so the index stays current after every sync.

Output: Two new modules (fts.rs, query.rs), one new migration (002_fts5.sql), and modifications to schema.rs (migration registration), sync.rs (FTS rebuild call), and lib.rs (module declarations).
</objective>

  <action>
    **Migration 002_fts5.sql** [FTS-01, FTS-02]:
    Create external-content FTS5 virtual table indexing message_content.text_content:
    ```sql
    CREATE VIRTUAL TABLE fts_message_content USING fts5(
        text_content,
        content='message_content',
        content_rowid='id',
        tokenize='unicode61'
    );
    ```
    Include a comment noting FTS-02 (file_operations content) is deferred to Phase 5 because the file_operations table does not exist yet. The search endpoint is designed to extend to file_operations when that table is created.

    **schema.rs** — Add ("002", include_str!("../migrations/002_fts5.sql")) to the MIGRATIONS array. Follow the existing pattern established in PAT-010.

    **fts.rs** [FTS-03] — Create new module with:

    1. `SearchResult` struct (Serialize, Debug): message_uuid, session_id, message_type, timestamp, block_type, snippet, rank (f64). This is the return type for search queries.

    2. `rebuild_fts_index(conn: &rusqlite::Connection) -> Result<(), rusqlite::Error>`:
       Execute `INSERT INTO fts_message_content(fts_message_content) VALUES('rebuild');`
       Log via tracing::info after successful rebuild.
       This is the external-content FTS5 rebuild command per SQLite docs.

    3. `search_messages(conn: &rusqlite::Connection, query: &str, limit: usize, offset: usize) -> Result<Vec<SearchResult>, rusqlite::Error>`:
       - Sanitize user input by wrapping in double quotes with internal quote escaping: `format!("\"{}\"", query.replace('"', "\"\""))`. This treats the query as a phrase match and prevents FTS5 syntax injection (see Research Pitfall 3).
       - SQL joins fts_message_content -> message_content -> messages to get session context.
       - Use `snippet(fts_message_content, 0, '>>>', '<<<', '...', 30)` for context snippets.
       - Use `bm25(fts_message_content)` for ranking. ORDER BY bm25 ascending (lower = better match, see Research Pitfall 2).
       - LIMIT ?2 OFFSET ?3 for pagination.
       - Use `rusqlite::params!` for the three parameters.

    Do NOT use triggers for FTS sync — rebuild-after-sync is the intended pattern (see Research Pattern 2).
    Do NOT create a content-storing FTS5 table (no `content=` omission) — external content avoids storage duplication (Research Anti-Pattern 1).
  </action>

  <action>
    **query.rs** — Create new module with all store-layer query functions that CLI subcommands will consume. All functions take `&rusqlite::Connection` and return `Result<Vec<T>, rusqlite::Error>` with Serialize+Debug result structs. Follow the existing patterns: collect results inside conn scope, return to caller for formatting (Research Anti-Pattern 5 — no println inside DB operations).

    1. `SessionSummary` struct + `list_sessions(conn, project: Option<&str>, after: Option<&str>, before: Option<&str>, limit: usize)` [CLI-04]:
       - Dynamic WHERE clause builder with parameterized queries.
       - project filter uses `s.project_path LIKE '%' || ?N || '%'` (substring match).
       - after/before filter on `s.first_seen_at`.
       - LEFT JOIN messages for message_count per session.
       - Subquery for primary model (first non-null model in session).
       - ORDER BY s.first_seen_at DESC.
       - Use `rusqlite::params_from_iter` with a `Vec<Box<dyn rusqlite::types::ToSql>>` for dynamic parameters.

    2. `MessageResult` struct + `query_messages(conn, session_id: Option<&str>, message_type: Option<&str>, model: Option<&str>, tool: Option<&str>, after: Option<&str>, before: Option<&str>, limit: usize)` [CLI-03]:
       - Dynamic WHERE clause builder (same pattern as list_sessions).
       - If tool filter is present, add `EXISTS (SELECT 1 FROM tool_executions te WHERE te.message_uuid = m.uuid AND te.tool_name = ?N)`.
       - SELECT uuid, session_id, type, timestamp, model, stop_reason from messages.
       - ORDER BY m.timestamp DESC, LIMIT.
       - Always returns JSON-serializable structs (the query subcommand always outputs JSON).

    3. `TokenStats` struct + `token_stats_by_model(conn)` and `token_stats_by_session(conn, session_id: Option<&str>)` [CLI-06]:
       - SQL aggregation (SUM, COUNT) — NOT in-memory aggregation (see Research Pitfall 4).
       - GROUP BY m.model for by-model, GROUP BY m.session_id for by-session.
       - Returns group_key, message_count, total_input_tokens, total_output_tokens, total_cache_read, total_cache_creation.

    4. `tool_frequency(conn) -> Vec<ToolStats>` [CLI-06]:
       - GROUP BY tool_name, COUNT invocations, SUM is_error.
       - ORDER BY invocations DESC.

    5. `model_breakdown(conn) -> Vec<ModelStats>` [CLI-06]:
       - COUNT messages by model, percentage of total.

    6. `version_history(conn) -> Vec<VersionEntry>` [CLI-08]:
       - SELECT DISTINCT version, MIN(timestamp), MAX(timestamp) FROM messages WHERE version IS NOT NULL GROUP BY version ORDER BY MIN(timestamp).

    7. `schema_drift_list(conn) -> Vec<DriftEntry>` [CLI-09]:
       - SELECT field_name, record_type, version, sample_value, first_seen_at, source_context FROM schema_drift_log ORDER BY first_seen_at DESC.

    8. `session_messages_for_export(conn, session_id: &str, batch_size: usize, offset: usize) -> Vec<ExportMessage>` [CLI-07]:
       - Batch-based message retrieval for export. Returns message fields + content blocks as nested data.
       - ExportMessage struct includes uuid, type, timestamp, model, content_blocks (Vec of content block data), token_usage (optional).
       - Load messages in batches to avoid OOM on large sessions (Research Pitfall 5).
       - For each message in batch, load its content blocks from message_content and optionally token_usage.

    **sync.rs** [CLI-02]:

This keeps the FTS index in sync after every bulk import. Only rebuild if at least one file was synced (result.files_synced > 0) to skip unnecessary work when nothing changed.

    **lib.rs**:
    Add `pub mod fts;` and `pub mod query;` declarations.
