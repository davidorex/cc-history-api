<objective>
Implement the file watcher subsystem: notify-based filesystem watching with custom per-file debounce, an async processing loop that triggers sync_file and emits all 4 SSE event types, version change detection, and periodic FTS rebuild.

Purpose: This is the core real-time ingestion capability. When the daemon is running and Claude Code writes to a session JSONL file, the watcher detects the change, syncs the new data into SQLite, and pushes events to all connected SSE clients. This eliminates the need for manual `sync` commands during active use.

Output: A fully operational live-ingestion pipeline where file changes flow from filesystem notification through sync_file into the database and out through SSE events to connected clients.
</objective>

1. Add `notify = { workspace = true }` to crates/server/Cargo.toml [dependencies].

2. Create crates/server/src/watcher.rs with the following components:

   a. **FileDebouncer struct** (WATCH-02):
      - Fields: `last_synced: HashMap<PathBuf, Instant>`, `debounce_duration: Duration`
      - `new(debounce_secs: u64) -> Self`
      - `should_sync(&mut self, path: &PathBuf) -> bool` — returns true if debounce_duration has elapsed since last sync of this path. Updates last_synced timestamp on true.
      - `prune_stale(&mut self, max_age: Duration)` — removes entries older than max_age from the HashMap to prevent unbounded memory growth (research Pitfall 5). Call periodically from the watcher loop (e.g., every 60 iterations or on a timer).

   b. **WatcherState struct**:
      - `debouncer: FileDebouncer`
      - `last_known_version: Option<String>` — for SSE-05 version change detection
      - `data_ingested_since_fts_rebuild: bool` — flag for deferred FTS rebuild
      - `last_fts_rebuild: Instant` — tracks time since last FTS rebuild

   c. **spawn_watcher function** (WATCH-01, WATCH-03):
      - Signature: `pub fn spawn_watcher(projects_dir: PathBuf, tx: tokio::sync::mpsc::Sender<notify::Result<notify::Event>>) -> notify::Result<()>`
      - Uses `std::thread::spawn` to create a blocking thread (NOT tokio::spawn_blocking — the watcher must live indefinitely)
      - Inside the thread: create `notify::recommended_watcher` with a callback that uses `tx.blocking_send(result)` — the `let _ =` prefix discards errors when the channel is full or closed
      - Call `watcher.watch(&projects_dir, notify::RecursiveMode::Recursive)`
      - After watch is established, call `std::thread::park()` to block the thread forever — the watcher is dropped if this function returns (research Pitfall 1)
      - Return Ok(()) from the outer function (the thread is spawned, watcher is running)

   d. **is_new_file async function** (for SSE-03 session detection):
      - Signature: `async fn is_new_file(conn: &tokio_rusqlite::Connection, file_path: &str) -> bool`
      - Queries `SELECT COUNT(*) FROM sync_metadata WHERE file_path = ?1`
      - Returns true if count == 0 (file not previously synced)
      - On query error, returns true (conservative: treat as new)

   e. **check_version_change async function** (SSE-05):
      - Signature: `async fn check_version_change(state: &mut WatcherState, conn: &tokio_rusqlite::Connection, session_id: &str, event_tx: &tokio::sync::broadcast::Sender<SseEvent>)`
      - Query: `SELECT version FROM sessions WHERE session_id = ?1`
      - Compare result to `state.last_known_version`
      - If different: update last_known_version, send SseEvent::VersionChanged with old and new versions
      - If query fails: log at debug level, do not emit event

   f. **watcher_loop async function** (WATCH-01, WATCH-02, WATCH-03, SSE-02, SSE-03, SSE-04, SSE-05):
      - Signature: `pub async fn watcher_loop(mut rx: tokio::sync::mpsc::Receiver<notify::Result<notify::Event>>, conn: tokio_rusqlite::Connection, event_tx: tokio::sync::broadcast::Sender<SseEvent>, cancel: tokio_util::sync::CancellationToken)`
      - Initialize WatcherState with FileDebouncer::new(2), last_known_version: None, data_ingested_since_fts_rebuild: false, last_fts_rebuild: Instant::now()
      - Main loop uses `tokio::select!` to race between:
        (a) `rx.recv()` — new filesystem event
        (b) `cancel.cancelled()` — shutdown signal (break the loop)
        (c) A 30-second `tokio::time::sleep` interval — for periodic FTS rebuild check
      - For filesystem events:
        - On `Err(e)`: log at warn level ("File watcher error: {}", e) and continue — graceful recovery per success criterion 4
        - On `Ok(event)`: iterate over event.paths, filtering to only `.jsonl` files (skip non-jsonl)
        - For each .jsonl path:
          - Call `debouncer.should_sync(&path)` — skip if debounce not elapsed
          - Extract session_id via `claude_history_store::sync::extract_session_id(path)` — skip if None
          - Check `is_new_file(&conn, &path_str)` BEFORE calling sync_file — this determines SSE-03
          - Call `claude_history_store::sync::sync_file(&conn, &path, &session_id).await`
          - On Ok(result) with result.records_synced > 0:
            - Emit SseEvent::RecordAdded with session_id, records_synced, file_path (SSE-02)
            - If was_new_file: emit SseEvent::SessionStarted with session_id (SSE-03)
            - If result.overflow_fields_logged > 0: emit SseEvent::SchemaDrift with new_fields count and session_id (SSE-04)
            - Call check_version_change for SSE-05
            - Set data_ingested_since_fts_rebuild = true
          - On Err(e): log at warn level ("Live sync failed for {}: {}", path, e) and continue — do not crash
      - For the 30-second timer branch:
        - If data_ingested_since_fts_rebuild is true AND at least 30 seconds since last_fts_rebuild:
          - Call `conn.call(|conn| claude_history_store::fts::rebuild_fts_index(conn)).await`
          - Reset flag and update last_fts_rebuild timestamp
          - Log at info level ("Periodic FTS rebuild complete")
          - If rebuild fails: log at warn level, do not crash
      - Every ~100 events or every minute (whichever comes first), call `state.debouncer.prune_stale(Duration::from_secs(600))` to prevent unbounded HashMap growth

   Import the `SseEvent` type from `crate::events::SseEvent`. Import sync and fts functions from `claude_history_store`.

3. Add `pub mod watcher;` to crates/server/src/lib.rs (or the appropriate module declaration location).

Important caveats from research:
- The notify watcher MUST live on a std::thread, NOT a tokio task — it needs to block indefinitely
- Use `blocking_send` in the notify callback, NOT `send().await` — the callback runs on a non-async thread
- Do NOT call `rebuild_fts_index` inside the per-file sync path — only on the 30-second timer
- The `let _ = event_tx.send(...)` pattern is intentional — if no SSE clients are connected, the event is silently dropped (no subscribers = no error, just returns Err with the unsent value)
