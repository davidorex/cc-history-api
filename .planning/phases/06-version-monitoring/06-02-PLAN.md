  <name>Task 1: Update drift.rs to track occurrence counts</name>
  <files>crates/store/src/drift.rs</files>
  <action>
Change the INSERT statement in `log_overflow()` from:
```rust
"INSERT OR IGNORE INTO schema_drift_log
 (field_name, record_type, version, sample_value, source_context)
 VALUES (?1, ?2, ?3, ?4, ?5)"
```
To:
```rust
"INSERT INTO schema_drift_log
 (field_name, record_type, version, sample_value, source_context, occurrence_count, last_seen_at)
 VALUES (?1, ?2, ?3, ?4, ?5, 1, datetime('now'))
 ON CONFLICT(field_name, record_type, version) DO UPDATE SET
   occurrence_count = schema_drift_log.occurrence_count + 1,
   last_seen_at = datetime('now')"
```

Update the return value logic: previously `changed > 0` meant a new entry was inserted. With ON CONFLICT DO UPDATE, `changed` will be 1 for both new inserts and updates. To distinguish new entries from updates, check if occurrence_count is 1 after the execute (or simply count all executions as `changed` and adjust the logging level — since the tracing is informational, counting updates as well is acceptable; the semantic change is documented in the function's doc comment).

Actually, the simplest correct approach: track new entries by attempting a SELECT before the INSERT to check if the row exists, OR accept that the return count includes updates. The latter is simpler and functionally equivalent — the `log_overflow` return value is used only for SchemaDrift SSE event emission and debug logging. Accept the change: `changed` now counts both inserts and updates.

Update the debug log message to reflect this: "Schema drift field observed" rather than "New schema drift field discovered" since it may be a re-observation.

Update the function doc comment to explain the ON CONFLICT behavior and occurrence_count tracking.

**Update tests:**
- `test_log_overflow_idempotent`: Update the assertion — the second call should now return 1 (not 0) because ON CONFLICT DO UPDATE returns 1 for each updated row. The test should verify that occurrence_count is 2 after the second call, and last_seen_at is updated, but the row count in schema_drift_log remains 1.
- Add a new test `test_log_overflow_occurrence_count`: Insert same field twice, verify occurrence_count = 2 and last_seen_at is not NULL.

**In `decompose_user()`**, after `insert_message()` (step 2) and before drift logging (step 5):

1. Extract known overflow keys into local variables:
```rust
let is_compact = r.overflow.get("isCompactSummary")
    .and_then(|v| v.as_bool())
    .unwrap_or(false);
let source_tool_id = r.overflow.get("sourceToolUseID")
    .and_then(|v| v.as_str())
    .map(|s| s.to_string());
```

2. Build extra_json from remaining overflow (excluding promoted keys):
```rust
let mut remaining = r.overflow.clone();
remaining.remove("isCompactSummary");
remaining.remove("sourceToolUseID");
let extra_json = if remaining.is_empty() {
    None
} else {
    Some(serde_json::to_string(&remaining)?)
};
```

3. UPDATE the message row with extracted values:
```rust
tx.execute(
    "UPDATE messages SET is_compact_summary = ?1, source_tool_use_id = ?2, extra_json = ?3
     WHERE uuid = ?4",
    rusqlite::params![is_compact as i32, source_tool_id, extra_json, r.base.uuid],
)?;
```

NOTE: This UPDATE happens after the INSERT OR IGNORE in insert_message. If the INSERT was a no-op (duplicate uuid), the UPDATE still runs on the existing row. This is intentional — re-sync can update these newly-promoted fields on existing rows.

4. When logging overflow for drift detection, log the ORIGINAL r.overflow (including isCompactSummary/sourceToolUseID) so drift detection still captures all overflow fields. The drift logger tracks what Claude Code is sending; the decomposer promotes known fields to columns.

**In `decompose_assistant()`**, apply the same pattern for extra_json from the assistant.message.overflow map. Compact summary fields (isCompactSummary, sourceToolUseID) appear on USER records, not assistant records, so only extra_json is relevant here:

1. Build extra_json from message overflow:
```rust
let msg_extra = if r.message.overflow.is_empty() {
    None
} else {
    Some(serde_json::to_string(&r.message.overflow)?)
};
```

2. Build extra_json from record-level overflow:
```rust
let record_extra = if r.overflow.is_empty() {
    None
} else {
    Some(serde_json::to_string(&r.overflow)?)
};
```

3. Merge both into a single extra_json (combine both overflow maps if both non-empty):
```rust
let combined_extra = match (record_extra, msg_extra) {
    (Some(r), Some(m)) => {
        // Merge both JSON objects
        let mut r_map: serde_json::Map<String, serde_json::Value> = serde_json::from_str(&r)?;
        let m_map: serde_json::Map<String, serde_json::Value> = serde_json::from_str(&m)?;
        r_map.extend(m_map);
        Some(serde_json::to_string(&r_map)?)
    }
    (Some(v), None) | (None, Some(v)) => Some(v),
    (None, None) => None,
};
```

4. UPDATE the message row:
```rust
tx.execute(
    "UPDATE messages SET extra_json = ?1 WHERE uuid = ?2",
    rusqlite::params![combined_extra, r.base.uuid],
)?;
```
