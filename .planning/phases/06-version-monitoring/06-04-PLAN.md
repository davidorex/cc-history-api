<task type="auto">
  <name>Task 1: Enhance /v1/schema/versions and /v1/schema/drift API handlers</name>
  <files>crates/server/src/api/schema.rs</files>
  <action>
**Update the `versions` handler:**

Add a query parameter struct:
```rust
#[derive(Debug, Deserialize)]
pub struct VersionsParams {
    /// When true, includes per-version field diffs (new fields, disappeared fields)
    pub diff: Option<bool>,
}
```

Update the handler signature to accept `Query(params): Query<VersionsParams>`.

When `diff` is Some(true): call `version_history_with_diff()` and return `Vec<VersionDiffEntry>`.
When `diff` is None or Some(false): call `version_history_enhanced()` and return `Vec<VersionHistoryEntry>`.

Since these are different response types, use a serde-tagged enum or `serde_json::Value` for the response. The simplest approach: use `Json<serde_json::Value>` as the return type and serialize the appropriate response. Alternatively, define an enum:
```rust
#[derive(Serialize)]
#[serde(untagged)]
enum VersionsResponse {
    Timeline(Vec<VersionHistoryEntry>),
    Diff(Vec<VersionDiffEntry>),
}
```

Update the re-export at the top to include the new query types:
```rust
use claude_history_store::query::{VersionHistoryEntry, VersionDiffEntry, VersionDriftGroup};
```

**Update the `drift` handler:**

Replace the existing implementation that calls `schema_drift_list()` with a call to `drift_by_version()`. The new handler returns `Vec<VersionDriftGroup>`.

Keep the `record_type` filter parameter — apply it in Rust post-retrieval by filtering groups to only include record_types matching the filter.

Keep the `limit` parameter — apply it by counting total fields across all groups and truncating when the limit is reached.

Update the response type from `Vec<DriftEntry>` to `Vec<VersionDriftGroup>`.
